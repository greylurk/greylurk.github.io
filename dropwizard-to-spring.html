Case Study:

Replacing Dropwizard with Spring

What's a Dropwizard?

  • Dropwizard is "a java framework for developing ops-friendly high performance RESTful web services."
  • Spring Framework "provides a comprehensive programming and configuration model for modern java-based enterprise applications."
  • Apache Camel is a "versatile open-source integration framework based on known Enterprise Integration Patterns."

Why leave Dropwizard?

  • We weren't really writing RESTful webservices
  • We were already using Camel JMS, which is built on Spring JMS
  • Springs support for termination protection is stronger
  • more future proof
  • Spring is an industry standard (Helps with Recruiting, Stack Overflow Searches)

Hurdles we faced?

  • sps-java-shared SSM support is Dropwizard specific. Customized Spring Cloud Connection components
  • DropWizard uses JDBI, but Spring uses JDBC Templates
  • Weblogic Connector in sps-java-shared is coupled to Dropwizard
  • Spring doesn't have native metrics, but micrometer is almost a drop in replacement.
  • Configuration moved a YAML format into a different YAML or .properties file

Continuing Hurdles

  • Spring Boot Autodiscovery feels like magic
  • Lack of institutional knowledge around Spring
